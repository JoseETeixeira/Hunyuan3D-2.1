Per-Model 3D Studio — Wire the New Frontend to the Backend

Current state of the Hunyuan3D-2.1 webapp backend, the new Next.js per-model contract, and the gap between them.

The new Next.js frontend is a clean per-model studio backed today only by an in-memory mock. The webapp backend already has the hard parts — Hunyuan shape gen, per-face AI paint (hyface), depth-aware reface, gpt-image-2 / Gemini synthesis — but exposes a flat, per-run, job-keyed API with no model identity and no mesh-free reference generation. The work is an orchestration + persistence layer, a step-2 reference generator, and a mode cleanup.

Backend: Hunyuan3D-2.1/webapp/server.py (100KB) + pipeline.py (50KB) · Frontend: Downloads/3-d-model-generation-workflow · Architecture change: REQUIRED

Step-2 reference generation — imperative dependency graph

Stage 1

front ← seed image

Stage 2 · cardinals

left, right, top, bottom, back ← front + seed

Stage 3 · corners

fl, fr ← front+left+right+top
bl, br ← back+left+right+top

Reuse

image_edit.edit_image() (OpenAI→Gemini wrapper) and the generate_elevations "grow the context list" pattern for cross-view consistency.

Gap

Only generate_elevations is mesh-free, but it is building-specific, skips bottom, and produces no corners. Every other view-gen path needs a rendered mesh — unusable pre-mesh. Need generic per-view prompts (orthographic, cartoonish, "only this view's faces", rotation desc) + tweak-prompt.

Architecture change — required

Stop before implementation — present options in Design

This adds a persistence layer (per-model registry), a new public REST surface with a Model aggregate + job→model contract, a mesh-free reference generator, an execution decision (network-bound gpt-image vs the single GPU queue), a deployment/origin change (serving the Next.js app same-origin as /api/*), and removes texture modes. Each needs validated trade-offs.

Open questions to resolve

  • Persistence: per-model JSON sidecar + index (recommended), SQLite, or extend gallery glob?
  • Where step-2 generation runs: through the GPU queue (blocks GPU) or a separate async path (recommended — gpt-image is not GPU)?
  • /texture/base semantics: shape + hyface using front reference only (rest via gap-fill, then reface)? Mock implies yes.
  • Reface inputs: reface a view using its approved step-2 reference (not a fresh upload)?
  • Deployment: FastAPI serves the built Next.js app vs proxy vs Next rewrites for same-origin /api/*.
  • Default-mode removal: drop Hunyuan PBR from API/UI but keep the pipeline method, or remove entirely?
